Engenharia de Dados Data Lake Modelagem

Como realizar a modelagem de dados para um Data Lake?

📅 Abril de 2026 ⏱ Leitura: ~22 min 🎓 Nível: Intermediário / Avançado
Resumo — A modelagem de dados para um Data Lake difere substancialmente das abordagens relacionais tradicionais. Enquanto bancos relacionais impõem estrutura no momento da escrita (schema-on-write), um Data Lake armazena dados em sua forma bruta e aplica estrutura somente durante a leitura (schema-on-read). Este artigo detalha as principais estratégias de modelagem — incluindo a Arquitetura Medallion (Bronze, Silver, Gold), particionamento, formatos colunares, catalogação de metadados e governança — fornecendo fundamentos teóricos, exemplos práticos em SQL/Python e um conjunto de boas práticas consolidadas para equipes de dados.
Seção 01

Introdução

A explosão do volume, velocidade e variedade de dados — os famosos "3 Vs" do Big Data — tornou os bancos de dados relacionais insuficientes para muitos cenários modernos de análise. Empresas passaram a armazenar petabytes de logs, eventos de IoT, feeds de redes sociais, imagens e documentos em repositórios centralizados chamados Data Lakes.

Entretanto, um Data Lake sem modelagem adequada rapidamente se transforma em um Data Swamp — um pântano de dados inutilizáveis, sem rastreabilidade e com baixíssima qualidade. A modelagem disciplinada é o que separa um repositório analítico valioso de um acúmulo desordenado de arquivos.

⚠️
Data Swamp: Estudos do Gartner indicam que até 85% dos projetos de Big Data falham principalmente por falta de governança e modelagem adequadas, resultando em dados que não podem ser encontrados, compreendidos ou confiados.

Este artigo apresenta um guia abrangente para modelagem de dados em Data Lakes, cobrindo desde os fundamentos conceituais até a implementação prática com ferramentas modernas como Apache Spark, Delta Lake, Apache Iceberg e AWS Glue Data Catalog.

Seção 02

O que é um Data Lake?

Um Data Lake é um repositório centralizado que permite armazenar todos os seus dados estruturados, semiestruturados e não estruturados em qualquer escala. Ao contrário de um Data Warehouse, os dados são armazenados em formato nativo — sem transformações prévias — e só são estruturados no momento da análise.

Data Lake vs Data Warehouse

Característica Data Warehouse Data Lake
Estrutura dos Dados Estruturado (schema rígido) Qualquer formato (raw)
Momento do Schema Schema-on-write Schema-on-read
Custo de Armazenamento Alto (storage proprietário) Baixo (object storage)
Usuários Analistas de negócios Engenheiros, cientistas de dados
Escalabilidade Vertical / limitada Horizontal / quase ilimitada
Agilidade Transformações antecipadas Ingestão rápida, transformação posterior
Exemplos Snowflake, Redshift, BigQuery S3 + Spark, ADLS + Delta Lake
💡
O conceito moderno de Data Lakehouse (Delta Lake, Apache Iceberg, Apache Hudi) combina o melhor dos dois mundos: armazenamento barato de um Data Lake com as capacidades ACID e de desempenho de um Data Warehouse.
Seção 03

Arquitetura Geral de um Data Lake

Independentemente da plataforma escolhida (AWS, Azure, GCP ou on-premises), um Data Lake bem estruturado possui camadas funcionais bem definidas que cobrem todo o ciclo de vida dos dados.

🔄 Camada de Ingestão
Batch (ETL), Streaming (Kafka, Kinesis), CDC (Debezium), APIs REST
↓
🗄️ Camada de Armazenamento
Object Storage (S3, ADLS, GCS) · Parquet, ORC, Delta, Iceberg
↓
⚙️ Camada de Processamento
Apache Spark, Flink, dbt · Transformações Bronze → Silver → Gold
↓
📊 Camada de Consumo
BI Tools, ML Models, APIs, Ad-hoc SQL (Athena, Presto, Trino)
🛡️ Transversal: Governança & Catálogo
Metadados, Linhagem, Qualidade de Dados, RBAC, Auditoria

Separação entre Computação e Armazenamento

Uma das decisões arquiteturais mais relevantes em Data Lakes modernos é a separação entre computação e armazenamento. Ao armazenar dados em object storage (como o Amazon S3) e escalar clusters de processamento (Spark, EMR, Databricks) de forma independente, as organizações eliminam o gargalo de recursos e otimizam custos drasticamente.

Seção 04

Arquitetura Medallion (Bronze · Silver · Gold)

A Arquitetura Medallion, popularizada pelo Databricks, é hoje o padrão de modelagem mais adotado em Data Lakes. Ela organiza os dados em três camadas progressivas de qualidade e refinamento, cada uma servindo a propósitos distintos.

🥉

Bronze

Dados brutos, exatamente como chegaram da fonte. Nenhuma transformação é aplicada. Serve como a "fonte de verdade" histórica e imutável.

🥈

Silver

Dados limpos, validados e conformados. Deduplicados, tipados corretamente e com regras de negócio básicas aplicadas. Prontos para exploração.

🥇

Gold

Dados agregados e modelados para consumo final. Tabelas dimensionais, métricas de negócio, features de ML. Alta performance de leitura.

Detalhamento da Camada Bronze

A camada Bronze armazena dados com fidelidade total à fonte. O objetivo é nunca perder dados — mesmo que defeituosos — para possibilitar reprocessamento futuro.

Python · PySpark 
# Ingestão Bronze: dados brutos sem transformação
from pyspark.sql import SparkSession
from pyspark.sql.functions import current_timestamp, input_file_name

spark = SparkSession.builder.appName("bronze-ingest").getOrCreate()

df = (
    spark.readStream
         .format("kafka")
         .option("kafka.bootstrap.servers", "broker:9092")
         .option("subscribe", "orders-topic")
         .load()
)

# Adiciona metadados de ingestão
df_bronze = df.withColumn("_ingested_at", current_timestamp()) \
              .withColumn("_source_file", input_file_name())

# Grava em Delta Lake na camada Bronze
(
    df_bronze.writeStream
             .format("delta")
             .outputMode("append")
             .option("checkpointLocation", "s3://datalake/checkpoints/orders")
             .start("s3://datalake/bronze/orders")
)

Detalhamento da Camada Silver

A camada Silver aplica transformações de limpeza, validação de tipos, eliminação de duplicatas e normalização de campos. Aqui as regras de negócio básicas são codificadas.

Python · PySpark 
# Transformação Bronze → Silver
from pyspark.sql.functions import col, to_timestamp, trim, upper
from delta.tables import DeltaTable

# Lê da camada Bronze
df_raw = spark.read.format("delta").load("s3://datalake/bronze/orders")

df_silver = (
    df_raw
    # Remover duplicatas
    .dropDuplicates(["order_id", "event_timestamp"])
    # Filtrar registros inválidos
    .filter(col("order_id").isNotNull() & col("amount") > 0)
    # Normalizar tipos
    .withColumn("order_date",  to_timestamp(col("order_date"), "yyyy-MM-dd HH:mm:ss"))
    .withColumn("customer_id", col("customer_id").cast("long"))
    .withColumn("status",      upper(trim(col("status"))))
    # Selecionar colunas relevantes
    .select("order_id", "customer_id", "order_date",
            "amount", "status", "_ingested_at")
)

# MERGE (upsert) para idempotência
silver_table = DeltaTable.forPath(spark, "s3://datalake/silver/orders")
silver_table.alias("target").merge(
    df_silver.alias("source"),
    "target.order_id = source.order_id"
).whenMatchedUpdateAll().whenNotMatchedInsertAll().execute()

Detalhamento da Camada Gold

A camada Gold contém dados prontos para consumo por ferramentas de BI, relatórios executivos e modelos de Machine Learning. Aqui tipicamente aplicamos modelagem dimensional (Star Schema) ou estruturas de features.

SQL · dbt 
-- models/gold/fct_orders_daily.sql
-- Tabela fato: vendas diárias agregadas por cliente

SELECT
    DATE_TRUNC('day', o.order_date)  AS order_day,
    c.segment                           AS customer_segment,
    c.region                            AS region,
    COUNT(DISTINCT o.order_id)          AS total_orders,
    SUM(o.amount)                       AS gross_revenue,
    AVG(o.amount)                       AS avg_order_value,
    COUNT(DISTINCT o.customer_id)       AS unique_customers
FROM {{ ref('silver_orders') }}      AS o
JOIN {{ ref('silver_customers') }}  AS c
    ON o.customer_id = c.customer_id
WHERE o.status = 'COMPLETED'
GROUP BY 1, 2, 3
ORDER BY order_day DESC
Seção 05

Schema-on-Read vs Schema-on-Write

Uma das decisões fundamentais na modelagem de um Data Lake é quando e como o schema (estrutura) dos dados é definido e aplicado.

✅ Schema-on-Read — Vantagens
  • Ingestão rápida sem pré-processamento
  • Flexibilidade para dados não estruturados
  • Permite exploração exploratória
  • Facilita onboarding de novas fontes
  • Suporta evoluções de schema retroativas
⚠️ Schema-on-Read — Riscos
  • Erros descobertos tardiamente (data swamp)
  • Custo computacional maior na leitura
  • Dificulta governança e catalogação
  • Pode gerar inconsistências entre consumidores
🧠
Abordagem híbrida recomendada: Use schema-on-read na camada Bronze (dados brutos) e aplique schema enforcement progressivo nas camadas Silver e Gold usando ferramentas como Delta Lake, que suporta tanto evolução de schema quanto sua aplicação obrigatória.

Schema Evolution no Delta Lake

Python · Delta Lake 
# Schema evolution: adicionar coluna sem recriar tabela
df_new = spark.read.format("delta").load("s3://datalake/silver/orders")

# Nova coluna adicionada pela fonte upstream
df_with_new_col = df_new.withColumn("discount_pct", col("discount") / col("amount"))

# mergeSchema=True: adiciona automaticamente novas colunas
(
    df_with_new_col.write
                   .format("delta")
                   .option("mergeSchema", "true")
                   .mode("append")
                   .save("s3://datalake/silver/orders")
)

# Enforcement: bloquear colunas inesperadas na Gold
spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "false")
Seção 06

Formatos de Armazenamento

A escolha do formato de arquivo tem impacto direto na performance de leitura, custo de armazenamento e capacidade de evolução do schema. Formatos colunares são amplamente preferidos em Data Lakes analíticos.

Formato Tipo Compressão Predicate Pushdown Schema Evolution ACID Ideal para
CSV Linha Baixa Não Não Não Ingestão inicial / interoperabilidade
JSON Linha Baixa Não Parcial Não APIs, semi-estruturado
Avro Linha Média Não Sim Não Streaming / Kafka
Parquet Colunar Alta Sim Parcial Não Analytics em geral
ORC Colunar Alta Sim Parcial Não Hive / Hudi
Delta Lake Colunar+Log Alta Sim Sim Sim Lakehouse moderno
Iceberg Colunar+Log Alta Sim Sim Sim Multi-engine / neutralidade
✅
Recomendação: Para novos projetos, prefira Delta Lake ou Apache Iceberg como formato padrão em todas as camadas. Eles oferecem transações ACID, evolução de schema, time travel e performance comparável a um Data Warehouse para workloads analíticos.
Seção 07

Estratégias de Particionamento

O particionamento é uma das técnicas mais impactantes para a performance em Data Lakes. Ele organiza os dados em subdiretórios físicos, permitindo que as engines de query pulem partições irrelevantes (partition pruning).

Particionamento por Data (mais comum)

Python · PySpark 
# Estrutura no S3 gerada pelo particionamento:
# s3://datalake/silver/orders/
#   year=2024/month=01/day=15/part-0001.parquet
#   year=2024/month=01/day=16/part-0001.parquet

df_silver.write \
         .format("delta") \
         .partitionBy("year", "month", "day") \
         .mode("overwrite") \
         .save("s3://datalake/silver/orders")

# Leitura eficiente: só lê a partição necessária
df = spark.read.format("delta") \
               .load("s3://datalake/silver/orders") \
               .filter(("year = 2024") & ("month = 1"))
# Spark lê apenas s3://datalake/silver/orders/year=2024/month=01/

Erros Comuns de Particionamento

🔥

Over-partitioning

Particionar por colunas de alta cardinalidade (como user_id) gera milhões de arquivos pequenos, degradando drasticamente a performance de listagem e leitura.

📦

Under-partitioning

Partições muito grandes impedem o partition pruning eficaz. Queries sempre lêem mais dados do que o necessário, aumentando custo e latência.

⚖️

Tamanho ideal de arquivo

Cada arquivo Parquet deve ter entre 128MB e 512MB. Use OPTIMIZE (Delta) ou REWRITE DATA FILES (Iceberg) periodicamente para compactar small files.

🔑

Chave de partição certa

A chave ideal é aquela usada com frequência nos filtros WHERE das queries de consumo. Analise os padrões de acesso antes de definir a estratégia.

SQL · Delta Lake 
-- Compactar small files e reordenar dados (Z-ordering)
-- Z-Order: organiza dados fisicamente para otimizar filtros multi-coluna
OPTIMIZE delta.`s3://datalake/silver/orders`
ZORDER BY (customer_id, order_date);

-- Remover arquivos antigos (vacuum)
VACUUM delta.`s3://datalake/silver/orders` RETAIN 168 HOURS;
Seção 08

Catálogo de Dados e Metadados

Sem catalogação, um Data Lake torna-se innavegável. O catálogo de dados é o componente que registra onde os dados estão, o que significam, quem os criou e qual é sua linhagem — permitindo que qualquer pessoa na organização encontre e compreenda os dados disponíveis.

Categorias de Metadados

Tipo Descrição Exemplos
Técnicos Schema, tipos, localização física, tamanho Glue Catalog, Hive Metastore
Operacionais Frequência de atualização, SLA, pipelines Airflow, dbt docs
Negócio Definições, dicionário de dados, owner Atlan, DataHub, Alation
Qualidade Perfis estatísticos, regras de validação Great Expectations, dbt tests
Linhagem Origem, transformações aplicadas, consumidores OpenLineage, Marquez

            "owner":          "data-engineering@empresa.com",
            "update_frequency": "daily",
        }
    }
)
Seção 09

Qualidade de Dados e Governança

Qualidade de dados não é um passo opcional — é um requisito arquitetural. Dados de baixa qualidade comprometem decisões de negócio, modelos de ML e a confiança dos consumidores no Data Lake.

Dimensões de Qualidade

✅

Completude

Percentual de valores não nulos em campos obrigatórios. Ex: order_id IS NOT NULL ≥ 99.9%

🎯

Unicidade

Ausência de duplicatas em chaves primárias e de negócio. Detectada com contagens distintas.

🔗

Consistência

Conformidade com regras de negócio entre tabelas. Ex: todo pedido deve ter um cliente válido.

⏰

Pontualidade

Os dados chegam dentro do SLA esperado. Monitorado por freshness checks automáticos.

Validação com Great Expectations

Python · Great Expectations 
import great_expectations as gx

context = gx.get_context()

# Define um conjunto de expectativas para a tabela Silver
suite = context.add_expectation_suite("silver_orders_suite")

validator = context.get_validator(
    batch_request=...,
    expectation_suite_name="silver_orders_suite"
)

# Completude
validator.expect_column_values_to_not_be_null("order_id")
validator.expect_column_values_to_not_be_null("customer_id")

# Unicidade
validator.expect_column_values_to_be_unique("order_id")

# Valores válidos
validator.expect_column_values_to_be_in_set(
    "status", ["PENDING", "PROCESSING", "COMPLETED", "CANCELLED"]
)

# Range numérico
validator.expect_column_values_to_be_between(
    "amount", min_value=0.01, max_value=1_000_000
)

# Freshness: dados não podem ser mais antigos que 25h
validator.expect_column_max_to_be_between(
    "order_date",
    min_value="now() - INTERVAL 25 HOURS"
)

results = validator.validate()
if not results["success"]:
    raise ValueError(f"Qualidade de dados reprovada: {results}")

Governança e Controle de Acesso

Um Data Lake multi-tenant exige controle fino de acesso por camada, domínio e sensibilidade dos dados. A abordagem recomendada é RBAC + ABAC (controle baseado em papéis combinado com atributos).

🔒
LGPD/GDPR: Dados pessoais identificáveis (PII) devem ser mascarados, pseudonimizados ou criptografados na camada Bronze. Use técnicas como tokenização ou data masking antes de propagar para camadas superiores. Nunca armazene dados sensíveis em texto claro nas camadas Silver e Gold sem controle explícito de acesso.
SQL · Unity Catalog (Databricks) 
-- Criar catálogo e schema com namespacing por camada
CREATE CATALOG IF NOT EXISTS production;
CREATE SCHEMA IF NOT EXISTS production.silver;
CREATE SCHEMA IF NOT EXISTS production.gold;

-- Conceder acesso granular
GRANT USE CATALOG ON CATALOG production TO `data-analysts`;
GRANT SELECT ON SCHEMA production.gold TO `data-analysts`;

-- Engenheiros têm acesso às camadas brutas
GRANT ALL PRIVILEGES ON SCHEMA production.silver TO `data-engineers`;

-- Column-level masking para dados PII
CREATE FUNCTION mask_email(email STRING)
RETURN CASE
    WHEN is_member('pii-access') THEN email
    ELSE CONCAT(LEFT(email, 2), '***@***.com')
END;
Seção 10

Boas Práticas de Modelagem

  1. Defina o domínio antes da ferramenta
    Organize os dados por domínio de negócio (orders, customers, products) antes de decidir tecnologias. A abordagem Data Mesh encapsula isso no conceito de "domínios como produtos de dados" com contratos bem definidos.
  2. Sempre adicione colunas de auditoria
    Toda tabela deve conter _ingested_at, _updated_at, _source_system e _pipeline_version. Essas colunas permitem depuração, reprocessamento e auditoria sem custo adicional de linhagem.
  3. Projete para idempotência
    Pipelines devem poder ser reexecutados sem duplicar dados. Use operações MERGE/UPSERT ao invés de INSERT simples. Isso é crítico para garantir exatamente-uma-vez (exactly-once) em sistemas de streaming.
  4. Documente contratos de dados (Data Contracts)
    Um Data Contract é um acordo formal entre produtor e consumidor definindo schema, SLA de atualização, regras de qualidade e responsáveis. Ferramentas como Soda e dbt contracts automatizam sua aplicação.
  5. Monitore e alerte proativamente
    Implemente freshness checks, alertas de volume anômalo e dashboards de qualidade antes de problemas chegarem aos consumidores finais. Ferramentas como Monte Carlo e Bigeye oferecem observabilidade nativa.
  6. Evite joins entre camadas diferentes
    Nunca realize joins entre tabelas Bronze e Gold em queries de produção. Cada camada deve ser auto-suficiente para seu propósito. Joins cruzados introduzem dependências difíceis de manter e gerenciar.
  7. Use Time Travel para debugging e rollback
    Delta Lake e Iceberg oferecem time travel nativo. Use-o para auditar estados históricos, comparar versões e reverter corrupções sem precisar de backups manuais onerosos.
  8. Trate o Data Lake como produto
    Aplique princípios de Product Management: defina usuários, colete feedback, meça SLAs e itere. Um Data Lake sem ownership claro e métricas de uso torna-se um custo sem retorno.
Seção 11

Conclusão

A modelagem de dados para um Data Lake é uma disciplina que combina princípios de engenharia de software, arquitetura de dados e governança corporativa. Ao contrário do que muitos imaginam, a flexibilidade de um Data Lake não elimina a necessidade de modelagem — ela a torna ainda mais crítica.

A Arquitetura Medallion (Bronze → Silver → Gold) estabelece uma progressão clara de qualidade e finalidade, enquanto formatos como Delta Lake e Apache Iceberg eliminam as limitações históricas de sistemas de arquivos distribuídos, adicionando transações ACID, evolução de schema e time travel.

O futuro aponta para o paradigma Data Lakehouse — onde a distinção entre Data Lake e Data Warehouse desaparece — e para o Data Mesh, que descentraliza a responsabilidade dos dados para os domínios de negócio, tratando dados como produtos de primeira classe.

🚀
Próximos passos recomendados: Comece com um projeto piloto usando a Arquitetura Medallion em uma única fonte de dados crítica. Estabeleça contratos de dados, implante validação de qualidade com Great Expectations ou dbt tests, e expanda gradualmente para outras fontes, mantendo sempre a governança como pilar central.
